# 后续演进路线

## 目的

本文档记录 `SLA 智能预诊助手` 从第一阶段纯前端 Demo 到后续目标架构的可能演进路线，明确每个阶段的目标、输入、输出和边界。

## 适用范围

适用于路线评审、后续需求讨论和阶段拆解。本文不代表任何后续阶段已确认上线，也不代表已接入真实客户、真实系统或真实数据。

## 正文

### 当前路线基线（CONFIRMED，2026-07-14）

1. `M1–M4`：现有 Vue、Spring Boot Mock API、Compose 工程化、架构/反馈/审计/权限骨架，均只代表已交付的 Mock 范围。
2. `一期契约门禁`：`/api/v2` OpenAPI 已完整定义方法、Schema、错误码、幂等、分页、异步任务状态和 v1→v2 映射，并于 2026-07-14 获人工批准。API v2 为 `APPROVED_FOR_IMPLEMENTATION`，一期实施为 `READY_FOR_IMPLEMENTATION`；当前已实现 AuthSession、知识上传/治理/发布、持久化 Precheck Session/Run、授权混合 Retrieval 与 Evidence、独立 Feedback、SubmissionContinuation 和结构化 AuditEvent。Evaluation、Metrics、Admin Reset 等未标记 `IMPLEMENTED` 的 v2 operation 仍未实现。
3. `一期本地完整纵向闭环`：在模拟、公开或脱敏数据边界内，覆盖本地身份与 RBAC、知识上传/解析/审核/发布、PostgreSQL/pgvector 混合检索、确定性预诊、多轮补充、独立反馈与模拟继续提交、审计及固定评估集。
4. `一期验收终点`：模拟继续提交已记录，反馈与审计已持久化，固定评估集达到工程门槛；不生成 SLA/工单，不联调真实 AIOps，不调用真实生成式模型。
5. `二期及以后`：真实身份、AIOps/ITSM、真实知识源、附件内容、生成式模型、生产治理等必须分别完成授权、契约、安全和数据评审后再进入路线。

下面的“第 0–7 阶段”为 2026-07-14 之前的历史拆分参考；凡与上述当前基线冲突之处，以当前基线及 `docs/project/scope.md` 为准，不再作为一期事实来源。

### 第 0 阶段：仓库清理与架构文档基线

目标：完成 `docs/` 分层目录、架构设计 docx 整合、实施计划、任务拆分、API / RAG / LLM / 知识库 / 集成 / 安全 / 观测 / 测试设计草案。

输入：现有 Demo、PRD、mock 数据、架构设计方案 docx 和协作规范。

输出：可评审的文档基线和后续任务拆分。

边界：不开发真实后端，不接真实接口，不使用真实客户数据。

### 第一阶段：纯前端 SLA 智能预诊 Demo

目标：验证 SLA 提交前 `智能预诊` 入口、loading、结果展示和人工审核提示。

输入：模拟 SLA 表单字段和前端 mock 数据。

输出：模拟参考资料、前置诊断结论、建议补充信息和操作提示。

边界：不接入真实后端、真实 RAG、真实大模型、真实 AIOps 或真实 SLA 系统。

### 技术 MVP M1（已完成）

目标：建立可本地运行的 Vue 前端与 Spring Boot Mock API 闭环，验证契约、结构化错误、自动化测试和人工降级能力。

输入：第一阶段 Demo 交互、技术 MVP 范围、OpenAPI 契约和 `模拟数据`。

输出：SLA 表单、`POST /api/v1/precheck`、确定性模拟预诊、结果面板、人工继续提交模拟、前后端行为测试和本地开发说明。于 2026/07/12 以 PR #3、merge commit `4da5765933906c79508a63e6fd4a200d2a1a593c` 完成交付。

边界：建议仅供人工审核，不作为最终根因、最终方案或正式复盘结论；SLA 提交及内容由人工确认。仅使用 `模拟数据`，未接真实数据库、RAG、模型、AIOps、ITSM、Wiki、历史 SLA、外部服务或生产环境。后续阶段尚未批准。

### 技术 MVP M2：本地工程化交付

目标：使现有 Mock 闭环可通过 Compose 重复构建、启动、健康检查和冒烟验收。

输入：M1 前后端、OpenAPI 唯一契约和 `模拟数据`。

输出：多阶段镜像、Nginx API 反代、仅暴露 `localhost:5173` 的 Compose、`GET /api/v1/health`、契约校验和 CI 容器冒烟。

边界：采用交付镜像而非容器热更新；不增加持久化、鉴权、附件、真实提交或生产部署，不接真实数据库、RAG、模型或外部服务。完成仅代表本地工程验收。

### 一期 PRD 评审：SLA 预诊助手可试点系统

目标：定义一期 `SLA 预诊助手` 的可试点系统需求，明确受控知识源、大模型辅助建议、人工审核和 AIOps 既有能力预留边界。

输入：SLA 表单字段、受控知识源范围、公司既有 AIOps 的历史 SLA 查询和 SLA 提交能力说明、人工审核要求。

输出：PRD 主文档、HTML 评审展示页、AIOps 历史 SLA 查询和 SLA 提交的产品接口要求。

边界：只定义产品需求和接口预留要求，不代表已接入真实 AIOps、真实 RAG、真实大模型、真实 Wiki 或真实历史 SLA。

### 第二阶段：可试点系统设计评审

目标：评审 API 草案、权限脱敏、审计、降级、知识审核和 ITSM / AIOps 集成字段。

输入：架构基线、PRD、实施计划和任务拆分。

输出：接口评审结论、安全评审结论和试点范围建议。

边界：仍不写真实后端代码，不提供真实接口地址或生产部署脚本。

### 第三阶段：受控知识库与 RAG 原型

目标：在授权、脱敏、可审计的数据条件下验证知识解析、切片、元数据、混合检索、重排序和评估指标。

输入：已授权、已脱敏、可审计的知识库条目。

输出：可追溯的资料命中结果和来源说明。

边界：只提供资料检索，不输出最终根因，不自动提交 SLA。

### 第四阶段：RAG 与大模型预诊

目标：基于可追溯引用，对命中资料进行摘要和信息补全建议。

输入：检索结果、用户表单字段和人工确认的上下文。

输出：摘要、补充问题、风险提示和可编辑建议。

边界：大模型输出必须包含依据来源、置信度、人工介入建议和待补充信息，不作为最终处理结论。

### 第五阶段：AIOps 数据辅助分析

目标：在授权范围内摘要 AIOps 指标，提供异常线索。

输入：已授权指标、告警摘要、时间窗口和组件信息。

输出：指标趋势、异常时间段、可能关联对象和待核验数据点。

边界：指标相关性不能被描述为最终根因，处理动作必须由人工确认。

### 第六阶段：ITSM / AIOps 集成与反馈闭环

目标：在员工人工确认后，将标题、描述、组件、版本、级别、影响范围、附件信息和预诊摘要提交到公司既有系统，并记录反馈指标。

输入：人工确认后的 SLA 表单、预诊摘要、附件信息和待补充信息确认状态。

输出：工单草稿或提交结果、失败提示、重试建议、审计记录和指标事件。

边界：不得自动提交 SLA，不得自动闭单；AI 能力不可用时不得阻塞人工处理。

### 第七阶段：智能复盘报告草稿

目标：在 SLA 处理完成后，生成可人工审核的复盘报告草稿。

输入：原始 SLA、处理记录、人工确认的根因、恢复结果和改进事项。

输出：事件摘要、时间线、处置过程、改进建议和知识沉淀候选内容。

边界：复盘报告必须由提交人和处理人确认后才能进入知识库沉淀流程，不得自动发布知识库。
