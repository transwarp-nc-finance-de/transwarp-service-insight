# Transwarp Service Insight 系统架构设计方案 v1.0

## 目的

本文档将 `Transwarp Service Insight 系统架构设计方案v1.0.docx` 整理为可维护的 Markdown 架构基线，用于后续架构评审、任务拆分和阶段演进规划。

## 适用范围

适用于 `SLA 智能预诊助手` 的目标架构讨论、接口草案设计、RAG / LLM / 知识库 / 集成 / 安全 / 观测 / 测试文档编写。本文描述目标架构和后续阶段能力，不代表当前纯前端 Demo 已经实现这些能力。

## 正文

### 当前仓库状态

当前仓库处于 Demo / PRD / 架构设计阶段，已包含纯前端静态 Demo、mock 数据、PRD、架构文档和任务拆分。当前不包含后端代码、真实接口接入、真实 RAG、真实大模型、真实 AIOps、真实 ITSM、真实 Wiki、真实历史 SLA 或数据库。

### 业务目标

借助 AI 能力在 SLA 工单提交源头进行智能预诊，降低基础类、重复类、开源社区类问题的工单提交数量，同时提升一线处理效率与闭环质量。后续目标是形成“客户自助、内部预诊、二线专家”的三级支撑体系，但该目标不代表当前阶段已经面向客户或合作伙伴开放。

### 核心设计理念

- RAG 为主导：通过受控知识库检索和可追溯引用生成建议，避免单纯依赖模型内部知识。
- 无侵入集成：通过后续标准接口或嵌入方式进入现有 ITSM / AIOps 流程，不改变工单主流程。
- 全链路安全可信：优先私有化部署，结合权限过滤、脱敏、提示注入防御、幻觉拦截和审计。
- 数据闭环自演进：通过埋点、反馈、知识复审和指标看板持续优化检索、知识和模型策略。

### 架构目标与原则

目标架构关注效率、性能、可用性、安全合规和可观测性。原始方案中的指标，如基础/重复类工单下降、预诊接口 P99 延迟、Top5 命中率和 99.9% 可用性，均属于后续试点或生产阶段目标，不作为当前 Demo 验收指标。

设计原则包括：无侵入集成、分层解耦、数据闭环、私有化优先、弹性降级和安全左移。

### 逻辑分层架构

目标架构可分为以下层次：

| 层次 | 目标职责 | 当前状态 |
| --- | --- | --- |
| 前端交互层 | 工单页预诊入口、结果面板、人工确认 | 当前仅有纯前端 Demo |
| 业务服务层 | 预诊编排、权限校验、反馈记录、降级控制 | 后续设计目标 |
| 检索增强层 | 查询改写、混合检索、重排序、缓存 | 后续设计目标 |
| 大模型生成层 | Prompt 管理、结构化输出、幻觉拦截、模型路由 | 后续设计目标 |
| 知识管理层 | 多源接入、解析切片、审核、发布、过期 | 后续设计目标 |
| 集成与观测层 | ITSM / AIOps 对接、埋点、指标、审计 | 后续设计目标 |

### 关键集成关系

- ITSM / AIOps：目标架构中通过受控接口或嵌入方式提供预诊，不阻塞原工单提交流程。
- 知识来源：目标架构中接入售后知识库、处理手册、相似案例、Wiki、内部培训素材和使用手册，且必须授权、脱敏、可审计。
- 反馈闭环：目标架构中通过埋点事件进入指标系统，驱动知识复审和模型策略调整。

### 查询改写与意图识别

目标设计将员工口语化、碎片化输入转换为规范可检索查询，并识别故障排查、操作指导、咨询等意图。设计包括术语纠错、同义词扩展、错误码识别、产品组件规范化、IP / 日期 / 版本等实体识别。原始输入必须保留用于审计，改写结果不得覆盖用户真实描述。

### 混合检索服务

目标设计采用关键词检索与向量检索结合，经过去重合并、分数归一化和 Cross-Encoder 重排序后返回结果。检索必须强制过滤 `status=APPROVED` 的知识条目。缓存设计可包含本地 L1 和分布式 L2，但当前仓库不实现缓存或真实检索。

### 大模型生成与幻觉拦截

目标设计通过生成管理器统一封装模型调用、Prompt 模板、结构化 JSON 输出和多模型路由。输出必须包含诊断摘要、引用来源、置信度、人工介入建议和待补充信息。

幻觉拦截分为生成前、生成中和生成后：检索分数不足时不生成；生成中限制模型只基于参考资料回答；生成后进行引用对齐、实体一致性和敏感信息过滤。高风险结果降级为仅展示检索列表。

### 知识库建设与管理

目标设计包括多源接入、文档解析、语义切片、OCR / 视频转写候选能力、元数据标注、向量化、索引同步、版本保鲜和审核发布。知识状态机为：

```text
DRAFT -> REVIEW -> APPROVED / REJECTED -> EXPIRED / DEPRECATED
```

只有 `APPROVED` 状态可进入线上检索；状态变更必须记录操作人、时间和原因。

### 大模型推理服务工程化

目标设计包括统一 API 网关、限流认证、负载均衡、弹性伸缩、多模型路由、A/B 测试、熔断降级和模型版本管理。这些均为后续工程化目标，当前仓库不创建推理服务、K8s、Helm 或 GPU 部署代码。

### 反馈闭环与效果度量

目标埋点事件包括 `pre_check_trigger`、`pre_check_result`、`pre_check_adopt`、`ticket_outcome`、`feedback_rating`。指标覆盖降量、效率、质量、检索命中、采纳率、幻觉拦截率和用户反馈。指标用于改进服务质量，不用于自动判责。

### ITSM / AIOps 集成方案

目标架构支持在工单创建页嵌入预诊面板，后续可保存 `pre_diagnosis_id`、`adoption_status`、`referenced_knowledge_ids` 等扩展字段。所有提交动作必须由用户人工确认，AI 不得自动提交 SLA、自动闭单或自动判责。

### 降级与容灾策略

核心原则：AI 服务不可用绝不能阻塞工单提交主流程。检索超时、向量库故障、LLM 推理失败、整个 AI 能力层不可用或缓存异常时，应允许用户继续人工提交，并记录降级事件用于后续排查。

### 提示注入防御与 AI 安全

目标设计包括输入过滤、指令与数据严格分离、输出内容安全检测、PII 过滤、多轮上下文防护和红队测试。模型不得根据用户注入指令改变系统规则，不得泄露系统提示词或未脱敏敏感信息。

### 技术选型参考

原始方案中提到 Qwen / DeepSeek、LangChain、自研胶水层、pgvector / Milvus、Elasticsearch、bge-reranker、Unstructured、PyMuPDF、Tesseract、Kafka、Redis、Prometheus、Grafana、ELK、Kubernetes、Nginx / Kong 等技术。本文将其作为目标架构参考，后续需结合公司现有平台、安全要求和试点规模单独评审。

### 部署与资源规划

原始方案中的 Kubernetes、GPU 节点、ES、数据库、Redis、Kafka、ClickHouse 等部署视图为后续目标架构参考，不代表当前仓库需要创建生产部署目录或脚本。

### 阶段演进路线

1. 当前阶段：纯前端 Demo、PRD 和架构设计文档。
2. 可试点系统设计：接口、权限、审计、知识源和集成方案评审。
3. 受控 RAG 原型：授权脱敏数据、混合检索、评估指标。
4. LLM 预诊：结构化输出、引用对齐、幻觉拦截和人工审核。
5. ITSM / AIOps 集成：人工确认提交、反馈闭环和指标看板。

### 差距说明

当前 Demo 与目标架构之间仍存在明显差距：没有真实知识源、真实检索、真实模型、真实权限、真实审计、真实埋点、真实接口和真实部署。后续任何真实接入都必须经过人工评审、安全评审和数据授权。
