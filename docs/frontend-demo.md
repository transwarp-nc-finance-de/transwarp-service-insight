# 前端 Demo 需求

## 目的

本文档说明 `Transwarp Service Insight` 一阶段 `SLA 预诊助手` 的纯前端 Demo 改造要求，用于演示 SLA 提交前的知识检索与预诊辅助能力。

## 适用范围

适用于 Demo 设计、前端原型评审和交互说明。本文只描述纯前端静态 Demo，不要求真实后端、不要求真实知识库、不要求真实大模型能力、不接入真实 AIOps 或真实 SLA 系统。所有页面字段、客户、集群、项目、问题描述、检索结果与链接均为模拟数据。

## 正文

你现在是一个前端工程师，请基于现有 AIOps SLA 报备页面，实现一个“知识库检索前置辅助”的纯前端 Demo。

### 一、需求背景

当前 SLA 报备页面中，用户填写问题类型、客户名称、集群名称、项目名称、产品组件、产品版本、问题级别、服务类型、问题概述、问题描述等信息后，点击“确定”提交 SLA。

现在希望在 SLA 正式提交前，提供一个轻量级的“知识库检索”能力，用于模拟系统根据用户填写的问题信息，检索 Wiki、知识库、历史问题、历史 SLA、内部培训素材和使用手册，给用户提供几条参考资料。

该 Demo 只验证前端交互效果，不要求真实后端、不要求真实知识库、不要求真实大模型能力。

### 二、核心目标

在 SLA 提交按钮旁边新增一个按钮：

按钮名称：`知识库检索` 或 `智能检索`

用户点击后，前端读取当前表单中的关键信息，例如：

1. ERP 客户名称
2. ERP 集群名称
3. ERP 项目名称
4. 产品组件
5. 产品版本
6. 问题级别
7. 问题概述
8. 问题描述

然后模拟一次知识库检索和提交前预诊流程。

### 三、交互流程

用户填写 SLA 表单
-> 点击“知识库检索”
-> 页面展示加载状态，例如“正在检索 Wiki、知识库、历史问题与 SLA 案例，请稍候...”
-> sleep 1.5 到 3 秒，模拟查询过程
-> 展示模拟检索结果
-> 用户查看参考资料
-> 如果资料有帮助，可以自行参考
-> 如果资料无帮助，用户仍然可以点击“确定”正常提交 SLA 拉研发处理

注意：

1. 知识库检索不能阻塞 SLA 提交。
2. 知识库检索不能替代研发处理。
3. 知识库检索结果只作为参考。
4. 用户始终可以继续提交 SLA。
5. 不要实现自动闭单。
6. 不要实现自动处理 SLA。
7. 不要改变原有 SLA 提交流程。

### 四、页面改造要求

#### 1. 按钮位置

在原有底部按钮区域中，通常已有：

```text
取消、确定
```

请在“确定”按钮旁边新增：

```text
知识库检索
```

推荐按钮顺序：

```text
取消、知识库检索、确定
```

也可以放在问题描述区域右上角，但优先放在提交按钮附近，体现“提交前辅助检索”。

#### 2. 按钮状态

点击“知识库检索”后：

1. 按钮进入 loading 状态。
2. 按钮文案变为“检索中...”。
3. 防止重复点击。
4. 检索完成后恢复可点击状态。

#### 3. 加载提示

加载期间展示以下提示之一：

```text
正在检索 Wiki、知识库、历史问题与 SLA 案例，请稍候...
```

或：

```text
正在根据当前问题信息进行知识库匹配...
```

可以使用弹窗、抽屉、卡片区域、骨架屏或 Spin 加载效果。

#### 4. 检索结果展示方式

检索完成后，在页面中展示一个结果面板。

结果面板标题：

```text
知识库检索结果
```

结果面板需要包含：

1. 检索关键词。
2. 检索范围。
3. 前置诊断结论。
4. 推荐参考资料。
5. 操作提示。
6. 依据来源、置信度、人工介入建议和待补充信息。

### 五、模拟检索逻辑

当前是纯前端 Demo，不调用任何真实接口。

请在前端本地维护一组 mock 数据，例如：

```js
const mockKnowledgeResults = [
  {
    type: "Wiki",
    title: "Foundation 审计日志归档功能说明（模拟数据）",

    url: "https://wiki.example.com/foundation-audit-archive"
  },
  {
    type: "历史问题",
    title: "Foundation 审计日志归档按钮不可用问题处理记录（模拟数据）",

  },
  {
    type: "SLA案例",
    title: "TDS 4.0.1 Foundation 组件归档功能异常（模拟数据）",
```

点击检索按钮后，从 mock 数据中随机返回 3 到 5 条结果即可。

不要请求真实接口。不要访问真实 Wiki。不要访问真实 SLA 系统。不要访问真实 AIOps 数据。

### 六、前置诊断展示要求

检索结果区域中需要明确展示：

```text
前置诊断结论：
当前检索结果仅提供相关参考资料，暂未发现可以直接解决该问题的明确方案。
如参考资料无法解决当前问题，请继续提交 SLA，由研发或相关责任人进一步处理。

依据来源：页面表单字段与前端模拟知识库数据。
置信度：中。
人工介入建议：由提交人确认是否继续提交 SLA，由处理人确认资料是否适用于当前环境。
待补充信息：真实环境日志、复现步骤、影响范围、权限与配置核查结果。
```

也可以根据表单内容生成更贴近当前问题的文案，但不得进行真实根因判断。

### 七、视觉风格要求

1. 保持现有 AIOps 页面风格。
2. 使用蓝色作为主色调。
3. 不要破坏原页面布局。
4. 不要大幅调整原有表单。
5. 新增内容应轻量、清晰、偏辅助性质。
6. 检索结果可以使用卡片、抽屉或弹窗展示。
7. 每条结果要有类型标签，例如 Wiki、历史问题、SLA案例、操作手册。
8. 结果卡片中需要有标题、简述和“查看详情”链接。
9. 链接必须为 mock 链接或仅做展示。

### 八、技术约束

1. 只做纯前端 Demo。
2. 不写后端。
3. 不调用真实接口。
4. 不接入真实 RAG。
5. 不接入真实大模型。
6. 不接入真实 AIOps。
7. 不修改真实 SLA 提交流程。
8. 不影响“确定”按钮原有逻辑。
9. 不新增数据库。
10. 不新增登录鉴权逻辑。
11. 所有数据使用前端 mock 数据，并标注为模拟数据。
12. 使用 setTimeout 或 Promise sleep 模拟接口等待。
13. 检索结果随机生成即可。
14. 代码尽量保持简单，方便 Demo 演示。

### 九、验收标准

完成后需要满足：

1. SLA 表单页面中新增“知识库检索”按钮。
2. 点击按钮后出现明显 loading 效果。
3. loading 持续 1.5 到 3 秒。
4. 检索完成后展示知识库检索结果。
5. 检索结果包含 3 到 5 条模拟资料。
6. 每条资料包含类型、标题、简述和链接。
7. 页面明确提示“检索结果仅供参考，无法解决时可继续提交 SLA”。
8. 页面展示依据来源、置信度、人工介入建议和待补充信息。
9. 原有“确定”提交按钮仍然可用。
10. 不调用任何真实接口。
11. 不影响原有页面功能。

### 十、特别说明

这个 Demo 的目标不是证明系统已经能够自动解决 SLA，而是演示：

1. SLA 提交前可以增加知识库检索入口。
2. 系统可以模拟基于问题描述检索 Wiki、历史问题、知识库、培训素材和使用手册。
3. 检索结果可以辅助用户自查。
4. 如果检索结果无帮助，用户仍然可以正常提交 SLA。
5. 该能力后续可以扩展为真实 RAG、大模型前置诊断和智能复盘能力，但必须经过范围确认、脱敏授权、权限设计和人工审核机制评审。
