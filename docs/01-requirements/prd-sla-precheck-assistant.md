# SLA 预诊助手一期 PRD

## 目的

本文档定义 `Transwarp Service Insight` 项目中一期 `SLA 预诊助手` 的产品需求，用于需求评审、范围确认、试点拆解和后续版本管理。

一期目标是在内部员工提交 SLA 前提供智能预诊辅助，提升提交信息完整度、降低基础类和重复类问题的低价值提交，并提高有效 SLA 的后续处理效率。

## 适用范围

本文适用于一期 `SLA 预诊助手` 的产品需求评审、交互设计、试点准备、验收标准确认和后续任务拆解。

本文仅定义产品能力和边界，不包含后端代码、接口协议、数据库设计、真实 API 地址、真实 RAG 接入、真实大模型接入或真实 AIOps 实现。涉及 AIOps 的内容仅为后续对接的产品接口要求，不代表当前已经完成真实系统接入。

当前状态：本文是下一阶段决策输入和产品设计草案，不代表一期试点已获批。仓库当前仍处于 `第一阶段：纯前端 SLA 智能预诊 Demo`，只有 Demo 验收关闭且范围、指标基线、权限、脱敏、授权、审计和人工审核责任通过评审后，才可进入一期可试点系统设计评审。

## 正文

### 知识来源配置与开放知识回答增补（第一阶段 Demo）

本增补仅描述纯前端交互和后续目标设计，不代表已接入真实配置中心、RAG、大模型、Wiki、历史 SLA、产品手册或互联网。

- Demo 顶部提供“知识来源配置（模拟）”，默认开启知识增强回答、Wiki、历史 SLA、产品手册、开源资料和模型通用知识降级；使用浏览器 `localStorage` 模拟管理员全局配置持久化。
- 配置变更从下一次预诊开始生效，每轮对话保存当时的策略快照，不追溯修改已有对话。
- RAG 是检索能力，不是知识来源。Wiki、历史 SLA、产品手册和开源资料是四类可独立启停的来源；RAG 总开关关闭时忽略四类来源开关。
- 内部资料命中时优先生成内部资料模式回答；内部资料无命中且开源资料开启时，允许使用模拟官方资料、发布说明、安全公告和上游 Issue/PR。所有链接均使用 `example.com`，不联网。
- 无资料命中且通用知识降级开启时，仅输出低置信度一般排查方向；降级关闭时只提示信息不足、待补充信息和人工继续提交 SLA。
- 输入区支持添加、预览、删除和随消息发送内置模拟截图、模拟日志，不读取用户本地文件，不保存未脱敏附件正文。
- 每次模型式输出必须展示回答模式、已启用来源、实际命中来源、依据来源、置信度、人工介入建议、待补充信息和非最终结论声明；诊断命令仅限只读操作。

后续真实系统的有效来源必须由“管理员已启用来源 ∩ 用户有权来源 ∩ `APPROVED` 知识 ∩ 查询过滤条件”计算。普通用户不得在预诊请求中扩大管理员允许范围。

### 1. 项目背景

`Transwarp Service Insight` 的项目主线是借助 AI 能力提升交付与服务效率。对交付和售后团队而言，SLA 的数量、质量和处理效率是最直观的服务协作指标之一。

当前 SLA 提交前常见问题包括：提交信息不完整、问题描述缺少关键环境信息、基础类问题重复进入 SLA 流程、开源社区类问题缺少已有资料参考、相似问题未被复用。这些问题会拉长首次理解、首次分派和首次响应耗时，也会增加研发、服务和交付之间的往返沟通。

一期 `SLA 预诊助手` 面向内部员工，在 SLA 提交源头提供受控知识源检索、历史经验参考、补充信息建议和人工确认提示。系统输出仅作为辅助建议，不作为最终根因、最终处理方案或正式复盘结论。

### 2. 问题定义

一期重点解决以下提交前问题：

(1) 信息不完整类：标题、描述、组件、版本、影响范围、日志和复现条件不足，导致处理人需要反复追问。

(2) 基础类问题：已有手册、Wiki 或培训资料可覆盖，但提交人未能在提交前快速找到资料。

(3) 重复类问题：历史相似案例或历史 SLA 已有处理经验，但提交前未被识别和复用。

(4) 开源社区类问题：问题可能与开源组件已知行为、配置限制或社区经验相关，需要先补充依据和影响判断。

(5) 分类不清类：提交人难以判断是否应该继续提交 SLA，以及应补充哪些信息后再提交。

### 3. 产品目标

(1) 提升 SLA 提交质量，尤其是字段完整度、问题上下文、影响范围和附件信息质量。

(2) 降低低价值提交，将基础类、重复类和资料可自助解决的问题在提交前尽量引导为自助解决或补充后再提交。

(3) 提高有效 SLA 处理效率，减少首次理解、首次分派和首次响应阶段的低效沟通。

(4) 建立可追溯的 AI 辅助输出格式，确保每次建议都包含依据来源、置信度、人工介入建议和待补充信息。

(5) 为后续接入公司既有 AIOps 系统的历史 SLA 查询和 SLA 提交能力预留产品接口要求。

### 4. 用户角色

| 角色 | 主要诉求 | 一期价值 |
| --- | --- | --- |
| 内部交付、售后、服务员工 | 在提交 SLA 前快速判断问题是否已有资料可参考，并补全必要信息。 | 降低重复检索成本，提高提交内容质量。 |
| SLA 处理人 | 收到更完整、更可理解的问题上下文。 | 减少追问，提高分派和响应效率。 |
| 知识库维护人员 | 发现高频问题、资料缺口和知识沉淀候选。 | 优化知识库覆盖范围和资料质量。 |
| 管理和运营人员 | 观察 SLA 质量、预诊命中率和效率改进趋势。 | 形成服务效率改进指标。 |

### 5. 一期功能范围

#### 5.1 SLA 提交前预诊入口

在员工填写 SLA 标题、描述、组件、版本、问题级别、影响范围等字段后，提供 `SLA 预诊` 入口。

预诊入口用于在提交前触发辅助分析，不阻断员工继续提交 SLA。员工可在阅读预诊建议后选择补充信息、自助处理或继续提交。

#### 5.2 检索意图生成

系统根据员工填写的字段生成检索意图，字段包括但不限于标题、问题描述、产品组件、产品版本、问题类型、影响范围、错误信息摘要、附件或日志摘要。

检索意图用于匹配受控知识源和后续可接入的脱敏历史 SLA 信息，不代表系统已经接入真实接口。

#### 5.3 受控知识源检索

一期优先面向售后知识库、Wiki、内部培训素材、使用手册、处理手册和相似案例摘要进行产品能力设计。

知识来源必须具备权限、脱敏、可追溯和可审计条件。所有命中结果必须展示来源类型、标题、摘要、更新时间或版本信息、适用范围和人工核验提示。

#### 5.4 预留 AIOps 历史 SLA 查询能力

公司既有 AIOps 系统具备历史 SLA 查询能力。一期 PRD 仅定义后续对接时的产品接口要求，不定义具体 API 协议、接口地址、鉴权实现或后端代码。

预诊助手需要预留按关键词、组件、版本、问题类型、时间范围、处理状态等条件查询脱敏历史 SLA 或复盘摘要的能力，并在结果中展示来源、相似点、差异点和复用限制。

#### 5.5 预留 AIOps SLA 提交能力

公司既有 AIOps 系统具备 SLA 提交能力。一期 PRD 仅定义提交前产品要求：员工人工确认后，可将标题、描述、组件、版本、级别、影响范围、附件信息和预诊摘要作为提交内容的一部分传递给 AIOps。

该能力必须坚持人工确认，不允许自动提交 SLA，不允许自动闭单，不允许绕过员工对内容准确性的审核。

#### 5.6 大模型辅助生成预诊建议

在受控知识来源和可追溯引用基础上，大模型可辅助生成问题摘要、可能原因方向、建议补充信息、相似资料解释、是否建议继续提交 SLA 和 SLA 草稿字段优化建议。

大模型输出必须包含依据来源、置信度、人工介入建议和待补充信息。输出不得写成最终根因、最终处理结论、正式复盘结论或自动判责依据。

### 6. 核心输出内容

每次预诊结果必须包含以下结构：

| 输出项 | 要求 |
| --- | --- |
| 预诊摘要 | 用简明语言概括当前问题和已识别上下文。 |
| 相似资料 | 展示知识库、Wiki、培训素材、手册或脱敏历史 SLA 摘要。 |
| 可能原因方向 | 只能表达为待核验方向，不得表达为最终根因。 |
| 建议补充信息 | 明确需要补充的字段、日志、截图、时间窗口、影响范围或复现步骤。 |
| 是否建议继续提交 SLA | 给出建议但不阻断，员工仍可人工确认后继续提交。 |
| 依据来源 | 列出命中的资料来源、历史摘要或输入字段依据。 |
| 置信度 | 使用高、中、低表达，并说明影响置信度的主要原因。 |
| 人工介入建议 | 明确哪些结论需要提交人、处理人或知识库维护人员确认。 |
| 待补充信息 | 列出进入正式 SLA 前仍缺失的信息。 |

#### 6.1 输出示例：模拟数据

以下为格式示例，全部为 `模拟数据`，不代表真实客户、真实 SLA、真实知识库或真实内部链接。

| 字段 | 示例 |
| --- | --- |
| 问题摘要 | 模拟数据：员工提交的 Foundation 审计日志归档功能不可用问题，当前描述缺少操作用户权限、后台日志和归档路径状态。 |
| 可能原因方向 | 模拟数据：可能与权限配置、归档路径状态、存储空间或版本限制有关，需进一步核验。 |
| 依据来源 | 模拟数据：Wiki 条目、处理手册摘要、脱敏历史 SLA 摘要。 |
| 置信度 | 中。当前已有组件和版本信息，但缺少日志、权限和影响范围。 |
| 人工介入建议 | 需要提交人补充信息，处理人确认是否进入 SLA，知识库维护人员后续评估是否沉淀条目。 |
| 待补充信息 | 操作用户、错误时间、后台日志、归档路径、存储空间、近期变更记录。 |

### 7. AIOps 对接产品接口要求

本节仅定义产品接口要求，不包含真实接口地址、字段协议、鉴权实现或后端代码。

#### 7.1 历史 SLA 查询要求

| 要求 | 说明 |
| --- | --- |
| 查询条件 | 支持按关键词、组件、版本、问题类型、时间范围、处理状态等条件查询。 |
| 数据边界 | 只能查询员工有权限访问的脱敏历史 SLA 或复盘摘要。 |
| 返回内容 | 返回标题、摘要、组件、版本、处理状态、相似点、差异点、复用限制和来源标识。 |
| 结果展示 | 必须提示历史资料仅供参考，不代表当前问题最终根因。 |
| 失败处理 | 查询失败不阻断员工人工提交 SLA，应提示稍后重试或继续人工提交。 |

#### 7.2 SLA 提交要求

| 要求 | 说明 |
| --- | --- |
| 提交触发 | 必须由员工人工确认触发，不允许自动提交。 |
| 提交内容 | 标题、描述、组件、版本、级别、影响范围、附件信息、预诊摘要、待补充信息确认状态。 |
| 草稿保留 | 提交失败时保留草稿，并提示重试或人工处理。 |
| 提示边界 | 提交前必须提示预诊结果仅为辅助建议，不代表最终处理结论。 |

#### 7.3 权限要求

按员工身份、角色权限、组织范围、服务职责和资料可访问范围进行控制。员工只能查看其权限范围内的知识资料、脱敏历史 SLA 摘要和预诊记录。

#### 7.4 审计要求

需要记录查询条件、命中来源、预诊结果摘要、置信度、提交人、提交时间、人工确认动作和最终提交动作。审计记录用于合规追溯和效果评估，不用于自动判责。

#### 7.5 失败处理要求

历史查询失败、大模型摘要失败或 AIOps 提交失败均不得阻断人工提交 SLA。系统应提示失败原因、保留草稿、允许重试，并保留人工提交路径。

### 8. 流程设计

1. 员工填写 SLA 标题、描述、组件、版本、级别和影响范围。
2. 员工点击 `SLA 预诊`。
3. 系统基于表单字段生成检索意图。
4. 系统检索受控知识源，并在后续可接入条件满足时查询脱敏历史 SLA 摘要。
5. 系统结合命中资料生成预诊建议。
6. 员工查看相似资料、可能原因方向、建议补充信息、依据来源和置信度。
7. 员工选择补充信息、自助处理或继续提交 SLA。
8. 员工人工确认后提交 SLA。
9. SLA 处理人基于更完整的上下文进行分派和处理。
10. 后续由知识库维护人员评估高频问题和资料缺口。

### 9. 非目标范围

一期明确不做以下事项：

- 不自动阻断 SLA 提交。
- 不自动提交 SLA。
- 不自动闭单。
- 不自动判责。
- 不输出最终根因。
- 不输出最终处理结论。
- 不直接修改生产配置。
- 不绕过人工审核。
- 不开放给客户或合作伙伴。
- 不使用未脱敏的真实客户、真实 SLA、真实日志或真实内部链接。
- 不编写后端代码、真实接口协议、真实数据库设计或真实系统接入实现。

二阶段 `智能工单助手` 仅作为后续方向，不属于一期已确认上线能力，也不代表已真实接入客户或合作伙伴场景。

### 10. 指标设计

一期优先衡量提交质量提升，SLA 数量下降仅作为间接收益，不作为强压指标。

| 指标 | 定义 | 价值 |
| --- | --- | --- |
| SLA 字段完整度提升 | 提交时标题、描述、组件、版本、影响范围、附件等字段完整度变化。 | 衡量提交质量改善。 |
| 预诊后补充信息比例 | 员工在预诊后补充字段、日志、截图或影响范围的比例。 | 衡量预诊建议是否推动信息完善。 |
| 命中资料占比 | 预诊命中知识资料或脱敏历史 SLA 摘要的占比。 | 衡量知识覆盖和检索效果。 |
| 自助解决或补充后提交比例 | 基础类、重复类问题被自助解决，或补充信息后再提交的比例。 | 衡量低价值提交治理效果。 |
| 有效 SLA 占比提升 | 提交后无需大量追问、可进入处理流程的 SLA 占比变化。 | 衡量整体提交质量。 |
| 首次分派耗时下降 | 从提交到首次分派的平均耗时变化。 | 衡量处理效率。 |
| 首次响应耗时下降 | 从提交到首次响应的平均耗时变化。 | 衡量服务响应效率。 |
| 问题理解耗时下降 | 处理人理解问题并明确下一步所需时间变化。 | 衡量上下文质量。 |

### 11. 风险与边界

| 风险 | 说明 | 控制方式 |
| --- | --- | --- |
| 知识源质量和时效性风险 | 过期或不完整资料可能导致建议偏差。 | 展示来源、更新时间、适用范围和人工核验提示。 |
| 历史 SLA 复用风险 | 历史问题与当前环境可能存在差异。 | 展示相似点、差异点和复用限制。 |
| 数据脱敏和权限风险 | 历史 SLA、日志和客户信息可能包含敏感内容。 | 仅使用授权、脱敏、可审计的数据摘要。 |
| 大模型误判风险 | 模型可能过度推断或遗漏关键条件。 | 强制输出置信度、依据来源、待补充信息和人工介入建议。 |
| AIOps 对接稳定性风险 | 查询或提交能力可能失败。 | 失败不阻断人工提交，保留草稿并支持重试。 |
| 组织流程风险 | 员工可能误把建议当成结论。 | 页面和文档持续强调辅助建议边界。 |

### 12. 验收标准

(1) PRD 明确一期 `SLA 预诊助手` 的背景、目标、角色、范围、非目标、指标和风险边界。

(2) 预诊输出结构包含依据来源、置信度、人工介入建议和待补充信息。

(3) AIOps 历史 SLA 查询和 SLA 提交只定义产品接口要求，不出现真实接口地址、协议细节或后端实现。

(4) 提交流程坚持 `建议但不阻断`，员工可人工确认后继续提交 SLA。

(5) 文档明确所有 AI 输出均需人工审核，不得作为最终根因或最终处理结论。

(6) 涉及示例时均标注为 `模拟数据`，不包含真实客户、真实日志、真实 SLA 或真实内部链接。

### 13. 待确认事项

(1) 一期试点范围：具体覆盖哪些内部团队、组件和问题类型。

(2) 知识源准入标准：哪些知识库、Wiki、培训素材和手册具备权限、脱敏和审计条件。

(3) AIOps 对接排期：历史 SLA 查询和 SLA 提交能力是否进入一期试点范围，还是仅保留产品预留。

(4) 指标基线：字段完整度、首次分派耗时、首次响应耗时和有效 SLA 占比的当前基线。

(5) 审计策略：预诊记录、人工确认动作和提交动作的保存周期和访问权限。

(6) 人工审核责任：提交人、处理人、知识库维护人员和运营人员的确认边界。

### 14. 后续方向

二阶段可进一步讨论 `智能工单助手`，面向 SLA 处理过程提供分派辅助、处理过程摘要、复盘草稿和知识沉淀建议。但该方向不属于一期范围，不代表已确认上线，也不代表已开放给客户或合作伙伴。
