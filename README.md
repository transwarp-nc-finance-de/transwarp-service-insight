# Transwarp Service Insight

## 目的

本文档用于说明 `Transwarp Service Insight` 项目的定位、业务背景、阶段路线、核心能力、Demo 范围和后续演进方向，帮助产品、运维、交付、研发和 AI Agent 在统一边界下协作。

## 适用范围

适用于项目立项、需求讨论、Demo 设计、文档评审和后续实施规划。当前阶段只交付规范化项目文档与 Demo 方案，不包含系统代码、真实接口接入、真实客户数据或自动化闭单能力。

## 项目简介

`Transwarp Service Insight` 是一个借助 AI 能力提升交付与服务效率的服务洞察项目。项目总目标不是替代人工处理 SLA 或工单，而是在提交前、处理中和复盘后，为相关人员提供信息补全、知识检索、AIOps 线索摘要、处理建议草稿、SLA 草稿和复盘报告草稿。

一阶段能力名称为 `SLA 预诊助手`，聚焦内部员工在提交 SLA 前进行预诊断。该阶段重点不是强压 SLA 数量，而是在提交源头改善问题质量，降低基础类、重复类、开源社区类问题的 SLA 提交数量，并提升有效 SLA 的处理效率。

所有大模型输出均属于辅助建议，必须展示依据来源、置信度、人工介入建议和待补充信息。最终结论、SLA 提交、工单流转、复盘确认和知识库沉淀均由人工审核确认。

## 背景

在复杂交付与服务场景中，SLA 问题往往存在以下挑战：

- 提交信息不完整，影响分派和排查效率。
- 基础类、重复类、开源社区类问题可能在提交前即可通过资料自查或补充信息澄清。
- 问题分类依赖经验，容易出现重复沟通。
- 售后知识库、处理手册、SLA 复盘数据、Wiki、内部培训素材、使用手册和平台指标分散在多个系统或材料中。
- 复盘内容依赖人工整理，知识沉淀不稳定。
- 处理建议若缺少依据来源，难以被业务、交付和运维人员信任。

本项目通过 RAG 知识检索、模拟 AIOps 指标摘要和大模型文本生成能力，辅助人工提升服务处理效率。

## 阶段路线

- 一阶段：`SLA 预诊助手`。面向内部员工提交 SLA 前的预诊断，基于售后知识库、AIOps 中 SLA 复盘数据、Wiki、内部培训素材、使用手册等构建模拟知识库，辅助补全问题信息、检索参考资料、判断是否需要提交 SLA 草稿。
- 二阶段：`智能工单助手`。面向客户与合作伙伴的一般性使用、咨询和常见问题，名称可后续再定；当前仅作为路线规划，不扩展为未经确认的生产上线方案。
- 三阶段：待定。暂不展开具体能力、接入范围或上线计划。

## 目标

- 在 SLA 提交前识别信息缺口，并生成补充问题清单。
- 根据问题描述辅助判断问题类型、影响范围和紧急程度。
- 检索知识库和相似案例，给出有依据的排查建议。
- 摘要 AIOps 指标，辅助判断是否存在资源、依赖或任务队列异常。
- 生成可人工编辑的 SLA 工单草稿和复盘报告草稿。
- 在人工审核通过后，沉淀可复用知识条目。

## 适用场景

- 内部员工提交 SLA 前，需要确认问题信息是否完整、是否已有可参考资料的场景。
- 批处理任务延迟、任务等待、调度异常等数据平台 SLA 场景。
- 数据同步失败、下游报表延迟、服务质量下降等需要跨角色协同的场景。
- 售后知识库、处理手册、SLA 复盘数据、Wiki、内部培训素材和使用手册可以被结构化管理的场景。
- 需要保留人工审核、人工决策和责任边界的企业服务流程。

## 不适用场景

- 要求系统自动判责、自动闭单或自动提交正式结论的场景。
- 缺少人工审核机制且要求大模型直接执行高风险动作的场景。
- 需要使用真实客户数据但尚未完成数据脱敏、授权和权限设计的场景。
- 要求在当前 Demo 阶段接入真实 AIOps、ITSM、监控、工单或知识库系统的场景。

## 核心能力

- SLA 预诊：检查 SLA 提交信息是否完整，识别缺失字段和风险点。
- 问题分类：基于描述和知识库辅助判断问题类型、影响范围和紧急程度。
- RAG 检索：从模拟售后知识库、模拟 SLA 复盘数据、模拟 Wiki、模拟内部培训素材和模拟使用手册中检索依据。
- AIOps 摘要：对模拟指标数据进行趋势、异常点和关联对象摘要。
- SLA 草稿生成：生成可人工编辑的 SLA 提交内容。
- 智能复盘：生成事件过程、原因分析、处置过程、改进建议和知识沉淀草稿。
- 人工审核：所有建议和报告必须由提交人、处理人或负责人确认后生效。

## 总体流程

1. 用户输入问题描述、影响范围、发生时间和已知现象。
2. `SLA 预诊助手` 进行信息完整性预诊，输出补充问题和风险提示。
3. 系统进行问题分类，并给出分类依据和置信度。
4. 系统检索模拟知识库与模拟历史案例，形成相似案例摘要。
5. 系统读取模拟 AIOps 数据，生成指标摘要和异常线索。
6. 系统输出排查步骤、处理建议和是否建议提交 SLA 的辅助判断。
7. 人工审核建议内容，并决定是否提交 SLA。
8. SLA 处理完成后，系统生成复盘报告草稿。
9. 提交人和处理人审核复盘内容，确认后再进入知识库沉淀流程。

## Demo 范围

当前 Demo 聚焦一个模拟场景：

> 模拟数据：凌晨跑批任务延迟，部分任务长时间等待，影响日报生成。

Demo 使用模拟售后知识库、模拟历史 SLA、模拟 SLA 复盘数据、模拟 Wiki、模拟内部培训素材、模拟使用手册和模拟 AIOps 指标。所有样例均为模拟数据，不代表真实客户、真实系统、真实历史 SLA、真实知识库或真实接口返回。

Demo 只验证文档化方案和交互闭环，不实现真实系统接入、不自动创建工单、不自动闭单、不自动写入生产知识库。

## 演进方向

- 阶段 1：确认总项目命名与一阶段 `SLA 预诊助手` 路线，整理模拟知识库结构、字段标准和 Demo 流程。
- 阶段 2：验证面向客户与合作伙伴的一般性使用和咨询类问题智能工单助手方向。
- 阶段 3：待定，基于前两阶段评审结果再定义范围。
