# 第一阶段关闭与下一阶段决策提示词

## 使用时间

2026/07/10

## 使用目的

用于固化 `SLA 智能预诊助手` 第一阶段纯前端 Demo 的技术验收成果，建立人工签字关闭流程，并补齐进入下一阶段设计评审前的范围、知识治理、人工责任和指标基线决策输入。

## 适用范围

适用于项目验收记录、进度与评审状态、下一阶段决策门禁、提示词归档和变更记录；不适用于后端代码、真实接口、真实 RAG、真实大模型、真实 AIOps、真实 Wiki、真实历史 SLA、数据库或生产系统接入。

## 提示词正文

### 总结

当前最优先事项不是开发 RAG 或后端，而是完成业务签字并正式关闭第一阶段。技术验收已“有条件通过”，下一步按“关闭验收 → 建立基线 → 决策评审”的顺序推进。

### 关键工作

1. 固化当前成果：审阅并提交本轮文档变更；排除用户已有的 `.gitignore` 修改；确认提示词重命名在 Git 中被识别为 rename。
2. 关闭第一阶段：由前端负责人和业务评审人在目标办公环境使用 Edge 完成人工冒烟；由产品负责人确认 Foundation 主场景及 4 类模拟资料口径；由项目负责人填写签字人、最终结论和关闭日期。未完成前保持“有条件通过，待业务签字关闭”。
3. 补齐下一阶段决策输入：明确仅限内部员工的试点范围、知识来源治理、AIOps 仅预留、提交人/处理人/知识维护人员/运营负责人的人工审核 RACI，以及使用获批、脱敏数据计算的四项指标基线。
4. 召开阶段决策评审：基于批准范围、基线、数据授权和人工责任决定“进入设计评审 / 暂缓 / 退回补充”；只有全部硬门禁通过才评审 API 草案、知识权限、安全护栏、审计字段和降级策略。

### 验收与边界

- Edge 人工冒烟覆盖页面加载、loading、初轮输出、4 类快捷补充、自由输入低置信度、继续提交、确定和取消。
- 每轮输出包含模拟数据边界、依据来源、置信度、人工介入建议和待补充信息。
- 第一阶段关闭记录包含明确结论、责任人、日期和未决问题状态。
- 项目进度、PRD、评审记录和 CHANGELOG 使用同一阶段状态。
- 本轮不新增或修改真实 API、数据结构或系统接口，不生成后端代码，不接入真实接口或未脱敏数据。
- AIOps 默认仅预留；真实 RAG、LLM、Wiki、历史 SLA 和数据库暂不实施。
- SLA 提交、分类、影响判断、处理建议和知识沉淀始终由人工确认。
