# SLA 智能预诊 Demo 范围收敛与重构提示词

## 使用时间

2026/07/03

## 使用目的

用于将 `Transwarp Service Insight` 收敛为第一阶段纯前端 `SLA 智能预诊助手` Demo，并统一目录、文案、模拟数据边界和变更记录规则。

## 适用范围

适用于项目文档、纯前端静态 Demo、mock 数据、演示说明和仓库治理文件的重构；不适用于后端代码、真实接口、真实 RAG、真实大模型、真实 AIOps、真实 Wiki、真实历史 SLA 或数据库接入。

## 提示词正文

你现在是一个资深前端工程师、项目重构工程师和技术文档工程师。请对当前 `transwarp-service-insight` 项目进行一次完整的范围收敛、目录重构、文案统一和变更记录规范化修改。

当前项目目标不是完整的 SLA 智能处理系统，而是第一阶段的 **SLA 智能预诊助手纯前端 Demo**。

请严格按照以下要求修改项目。

# 一、项目当前定位

项目名称：

```
transwarp-service-insight
```

项目展示名称：

```
SLA 智能预诊助手
```

当前阶段目标：

在现有 AIOps SLA 报备页面中，新增一个“智能预诊”按钮。用户在提交 SLA 前，可以点击该按钮，系统基于当前表单信息，模拟检索 Wiki、历史问题、历史 SLA、操作手册和产品文档，并展示若干参考资料、前置诊断结论和建议补充信息。

如果检索结果对当前问题没有帮助，用户仍然可以继续点击“确定”提交 SLA，由研发或相关责任人进一步处理。

# 二、当前阶段范围

## 当前阶段只做

1. 纯前端静态 Demo。
2. SLA 表单页面新增“智能预诊”按钮。
3. 点击“智能预诊”按钮后展示 loading 效果。
4. 使用 `setTimeout` 或 `Promise sleep` 模拟检索耗时。
5. 使用前端 mock 数据模拟 Wiki、历史问题、历史 SLA、操作手册和产品文档。

8. 展示建议用户补充的信息清单。
9. 明确提示用户：如果资料无帮助，可继续提交 SLA。
10. 保留原有 SLA 提交流程，不阻塞用户提交。

# 六、前端 Demo 修改要求

请修改或完善：

```text
prototypes/sla-precheck-demo.html
```

## 1. 页面目标

模拟 AIOps SLA 报备页面，在原有“取消”“确定”按钮附近新增“智能预诊”按钮。

推荐按钮顺序：

```text
取消    智能预诊    确定
```

## 2. 点击交互

用户点击“智能预诊”后：

1. 按钮进入 loading 状态。
2. 按钮文案变为“预诊中...”。
3. 禁止重复点击。
4. 展示加载提示：

```text
正在检索 Wiki、历史问题、历史 SLA、操作手册和产品文档，请稍候...
```

1. 等待 1.5 到 3 秒。
2. 展示“智能预诊结果”。

## 3. 智能预诊结果内容

结果面板或弹窗中必须包含：

```text
智能预诊结果

检索关键词：
Foundation、TDS-4.0.1、审计日志归档、功能不可用

检索范围：
Wiki、历史问题、历史 SLA、操作手册、产品文档

前置诊断结论：
当前仅基于模拟知识库返回相关资料，暂未发现可以直接解决该问题的明确方案。
检索结果仅供参考，不代表最终根因或处理结论。
如参考资料无法解决当前问题，请继续提交 SLA，由研发或相关责任人进一步处理。

建议补充信息：
1. 是否所有用户均无法使用归档功能？
2. 当前操作用户是否具备审计管理权限？
3. 是否有后台服务报错日志？
4. 是否近期升级或变更过配置？
5. 是否存在归档路径、存储空间或权限异常？

推荐参考资料：
1. [Wiki] Foundation 审计日志归档功能说明
2. [历史问题] Foundation 审计日志归档按钮不可用问题处理记录
3. [SLA 案例] TDS-4.0.1 Foundation 组件归档功能异常
4. [操作手册] Foundation 组件常见问题排查手册

操作提示：
如果以上资料不能解决当前问题，请继续提交 SLA。
```

## 1. CHANGELOG.md 目的

`CHANGELOG.md` 用于记录每次有意义的项目变更，包括需求调整、文档修改、原型修改、目录重构、交互优化等。

## 2. 记录规则

必须写入以下规则：

```markdown
# 变更记录

## 记录规则

1. 按日期倒序记录，最新日期放在最上方。
2. 日期格式统一为 `YYYY/MM/DD`，例如 `2026/07/03`。
3. 同一天的多条变更必须合并到同一个日期标题下。
4. 不得重复创建相同日期标题。
5. 每条变更使用编号列表：`(1)`、`(2)`、`(3)`。
6. 每条变更尽量说明“改了什么”和“为什么改”。
7. 只记录有意义的项目变更，不记录临时调试、无关格式调整或个人操作。
8. 涉及真实客户、真实 SLA、真实日志或敏感信息时，必须先脱敏再记录。
9. 如果只是纯前端 Demo 或模拟数据变更，需要明确标注“模拟数据”或“Demo”。
```

## 3. 记录格式

必须使用这种格式：

```markdown
## 变更记录

### 2026/07/03

(1) 明确项目第一阶段定位为 `SLA 智能预诊助手`，聚焦 SLA 提交前的智能预诊与知识库参考资料推荐。

(2) 明确当前阶段以纯前端 Demo 为主，不接入真实后端、真实 RAG、真实大模型、真实 AIOps 或真实 SLA 系统。

(3) 规划新增“智能预诊”按钮，用于模拟根据 SLA 表单信息检索 Wiki、历史问题、历史 SLA、操作手册和产品文档。

(4) 约定智能预诊结果仅作为辅助参考，不作为最终根因、最终处理方案或自动闭单依据。
```

禁止使用这种格式：

```markdown
2026/07/03: 修改 a
2026/07/03: 优化 b
```

必须合并为：

```markdown
### 2026/07/03

(1) 修改 a。

(2) 优化 b。
```

# 十七、提交和变更记录要求

完成修改后，请更新 `CHANGELOG.md`，在当天日期下追加本次变更。

如果当天已经存在日期标题，例如：

```markdown
### 2026/07/03
```

不要新建第二个 `### 2026/07/03`，而是在原日期下继续追加编号。

本次变更可以记录为：

```markdown
(5) 重构项目目录结构，将前端 Demo 原型移动到 `prototypes/sla-precheck-demo.html`。

(6) 统一功能文案，将“知识库检索”调整为“智能预诊”。

(7) 新增 `CHANGELOG.md`，用于按日期合并记录项目变更内容。

(8) 更新 `AGENTS.md`，明确当前阶段允许纯前端 Demo，禁止后端、真实接口、真实 RAG、真实大模型和真实 AIOps 接入。

(9) 更新项目文档，将第一阶段范围收敛为纯前端 SLA 智能预诊 Demo。
```

# 十八、输出要求

修改完成后，请输出：

1. 修改了哪些文件。
2. 新增了哪些文件。
3. 移动了哪些文件。
4. 删除或建议删除哪些文件。
5. 当前项目目录结构。
6. 本次变更记录写入了 `CHANGELOG.md` 的哪些内容。
7. 是否仍存在需要人工确认的问题。

# 十九、特别约束

请严格遵守：

1. 不要生成后端代码。
2. 不要调用真实接口。
3. 不要引入数据库。
4. 不要接入真实 RAG。
5. 不要接入真实大模型。
6. 不要接入真实 AIOps。
7. 不要使用真实客户、真实 SLA、真实历史问题、真实日志或真实内部链接。
8. 所有示例数据必须明确标注为“模拟数据”。
9. 不要设计自动提交 SLA。
10. 不要设计自动闭单。
11. 不要让智能预诊结果阻塞 SLA 提交。
12. 不要把智能预诊结果描述为最终根因或最终处理结论。
13. 文档必须使用中文。
14. Markdown 文件格式要清晰，适合放入项目仓库。
15. HTML 原型只允许作为纯前端 Demo 使用。
